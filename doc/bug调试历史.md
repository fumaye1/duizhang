## 1) 2026-03-06：多仓汇总上传 `.xls` 报错缺少 xlrd

- 模块/入口：多仓汇总上传对账结果文件（`app.py` → `file_uploader_multi_block`）
- 现象：
  - Streamlit 页面直接报错：`ImportError: Missing optional dependency 'xlrd'... for xls Excel support`
  - 堆栈指向：`pd.ExcelFile(uploaded)` 初始化 reader 时触发
- 原因：
  - 用户上传的是旧版 Excel `.xls` 文件
  - pandas 读取 `.xls` 需要可选依赖 `xlrd>=2.0.1`，而环境未安装
- 解决办法：
  1. 依赖层：在 [requirements.txt](requirements.txt) 增加 `xlrd>=2.0.1`
  2. 代码层：在 [app.py](app.py) 增加 `_safe_excel_file()`，捕获 `ImportError`：
     - 若为 `.xls` 且缺依赖：页面提示可操作方案（安装 xlrd 或另存为 `.xlsx`）
     - 多文件上传时：跳过有问题文件并给 warning，避免整个页面崩溃
- 如何避免/加固：
  - 发布前检查：用“示例数据 + .xls”做一次冒烟测试
  - 产品提示：在上传控件旁提示优先使用 `.xlsx`（兼容性更稳定）
  - 依赖治理：将“可选依赖”也纳入 requirements，避免客户环境差异

### 1.1) 2026-03-06：安装 xlrd 后仍报 XLRDError（文件不是有效 xls）

- 模块/入口：同上
- 现象：
  - `xlrd.biffh.XLRDError: Can't find workbook in OLE2 compound document`
- 原因：
  - 文件扩展名虽为 `.xls`，但内容并不是合法的 xls 工作簿：
    - 常见于“把 .xlsx 改后缀成 .xls”、WPS/系统导出异常、文件损坏、或上传的其实是HTML/其他格式伪装成Excel。
- 解决办法：
  - 在 [app.py](app.py) 上传读取中增加“按文件头识别 xls/xlsx 并选用正确引擎”，并捕获 `XLRDError` 给出明确指引；多文件上传时跳过坏文件。
  - 业务侧处理：用Excel/WPS重新“另存为 .xlsx”或从源系统重新导出后上传。
- 如何避免/加固：
  - 不要通过“改后缀名”转换格式；一律用“另存为/导出为 .xlsx”
  - 若来源系统提供多种导出格式，优先选择 `.xlsx`
  - 可在导出流程中加入校验：文件大小异常小（如几KB）或打开即报错的文件不要用于对账

## 2) 2026-03：终端误入 Python REPL 导致命令报 SyntaxError

- 模块/入口：本地验证编译/运行时
- 现象：
  - 运行类似 `py -m py_compile ...` 时出现 `SyntaxError: invalid syntax`
- 原因：
  - 误进入 Python 交互式 REPL，把 shell 命令当 Python 语句执行
- 解决办法：
  - 退出 REPL（`exit()`/`Ctrl+Z` 回车），在 PowerShell/CMD 中执行命令
- 如何避免/加固：
  - 看到提示符 `>>>` 先退出再执行
  - 建议使用统一脚本：例如 `scripts/fix_python_env.ps1` + 固化检查命令

## 3) 2026-03：某 Tab 早退导致后续页签不渲染

- 模块/入口：Streamlit 多 Tab 页面（`app.py`）
- 现象：
  - 点击某个页签后，后面的页签（如新增模块）不显示/不渲染
- 原因：
  - 在某个 Tab 代码块中使用了 `st.stop()` 或在主函数早 `return`，导致整次脚本执行被中断
- 解决办法：
  - 将“早退/停止”改为局部提示与条件分支，不中断整个 `main()`
- 如何避免/加固：
  - 约定：Tab 内不要 `st.stop()` 影响全局；需要阻断只阻断当前 Tab 的计算
  - 做一次“空输入遍历所有 Tab”的冒烟测试

## 4) 2026-03：多云仓账单关联串账风险（运单号重复）

- 模块/入口：对账主流程账单关联（`reconcile.py`）
- 现象：
  - 不同云仓存在相同 `物流单号` 时，账单金额可能关联错行，导致差异异常
- 原因：
  - 账单关联键仅使用 `物流单号`，在多仓场景并不唯一
- 解决办法：
  - 将账单聚合与关联键升级为 `物流单号 + 云仓`
  - 账单聚合口径：同键费用求和、计费重量取 max
- 如何避免/加固：
  - 规则：凡是“跨仓合并/对账”都必须使用复合键
  - 质量检查：导入后提示“同运单号跨仓重复”的统计

## 5) 2026-03：资费表结构升级导致字段映射不一致/匹配失败

- 模块/入口：资费匹配与运费计算（`reconcile.py` + `app.py` 资费映射UI）
- 现象：
  - 资费未匹配、或运费计算不符合预期
- 原因：
  - 资费表从“单字段快递费”升级为 `首重/首费/续重/续费` + `重量段结束=0` 超重规则行
  - 若 UI 映射/字段校验仍沿用旧字段，会造成缺列或取值为空
- 解决办法：
  - `reconcile.py`：增加资费 schema 兼容与兜底映射（旧字段映射到新字段）
  - `app.py`：资费表映射改为新必填字段集合，避免误选
- 如何避免/加固：
  - 文档同步：README/PRD/示例数据保持一致（避免用户按旧模板导入）
  - 回归样例：加入“超重触发0段”的固定样例

## 6) 2026-03：大表导出 Excel 行数限制/单 Sheet 超限

- 模块/入口：多仓汇总导出（`app.py` 导出逻辑）
- 现象：
  - 导出失败或生成文件不可用（单 Sheet 行数过大）
- 原因：
  - Excel 单 sheet 行数存在上限，且写入过程内存压力大
- 解决办法：
  - 按阈值（默认 80 万，可配置）触发分 Sheet
  - 先按 `云仓` 拆分；单仓仍超阈值则继续切片 `云仓_1/云仓_2...`
  - 处理 sheet 名合法与唯一
- 如何避免/加固：
  - 导出前给出行数预估与拆分提示
  - 超大表建议优先导出按仓拆分文件（如后续需要）

  ## 7) 2026-03-06：多仓字段映射交互复杂/容易配错 → 改为“仅命中检查 + 不命中禁止生成”

  - 模块/入口：多仓汇总与回冲（`📦 多仓汇总与回冲` Tab）
  - 现象：
    - 多来源结果表列名“看起来很规范但不完全一致”，逐字段下拉/批量映射交互依然成本高。
    - 手动映射容易出现“选错列/漏选列/重复映射”，后续导出字段含义不一致。
  - 原因：
    - 多仓汇总属于“批量导出/复核”场景，最稳妥的方式是强制字段命名一致，避免工具内部隐式清洗和人为映射。
    - 如果允许在工具内做“同义词清洗/手动映射”，会引入不可见的口径差异和复核风险。
  - 解决办法（当前口径）：
    - 上传前：提示用户在源表中**手动**把列名改为规范字段名。
    - 上传后：工具仅做“命中检查”（列名同名即命中），并提示每个文件/Sheet 的未命中字段。
    - 未全部命中：直接阻止生成（汇总/回冲均如此），引导用户修改源表再上传。
  - 如何避免/加固：
    - 将“规范字段列表”沉淀到《示例数据.md》与交付模板；要求外部系统导出的列名按模板统一。
    - 真实交付时建议在客户侧形成“导出→列名规范化→上传”的固定动作。

  ## 8) 2026-03-06：多仓上传/字段检查慢（读取了整张明细）→ 分阶段懒解析 + 只读表头/只读所需列

  - 模块/入口：多仓汇总与回冲上传区域
  - 现象：
    - 上传后页面响应慢，尤其是“大工作簿 + 多Sheet + 大行数”时明显卡顿。
  - 原因：
    - 为了拿到列名进行命中检查，若直接 `pd.read_excel()` 会读取整张表，I/O 和解析开销很大。
  - 解决办法：
    - 分阶段处理：
      1) 上传阶段：仅解析工作簿的 Sheet 名（`pd.ExcelFile(...).sheet_names`）。
      2) 命中检查：仅读取表头（`nrows=0`）获取列名。
      3) 点击“生成”后：才读取明细数据，并用 `usecols=所选字段` 只读取需要的列。
    - 同时增加进度显示：表头检查进度、生成时读取明细进度。
  - 如何避免/加固：
    - 大文件建议优先拆分为多个结果文件或减少不必要字段。
    - 若字段较多，生成阶段坚持使用 `usecols` 避免全量读取。

  ## 9) 2026-03-06：一个工作簿需要选择多个 Sheet → 上传控件支持多选 Sheet

  - 模块/入口：多仓汇总/回冲的“对账结果文件（可多选）/新复核结果表（可多选）”
  - 现象：
    - 业务常见一个工作簿里按云仓/月份/批次拆成多个 Sheet，需要一次性选中多个 Sheet 参与汇总。
  - 解决办法：
    - Sheet 选择从单选改为多选；每个被选中的 Sheet 都作为独立输入项参与命中检查与生成。
  - 如何避免/加固：
    - 在文件命名或 Sheet 命名中加入批次信息（如：`华东仓_2026-03_1`），便于定位未命中字段。

  ## 10) 2026-03-06：扩展名与真实格式不一致导致读取失败 → 通过文件头识别引擎并贯穿使用

  - 模块/入口：所有 Excel 上传（包含多仓模块）
  - 现象：
    - 文件后缀为 `.xlsx` 但实际是 OLE2（xls），或反过来，导致读取失败。
  - 解决办法：
    - 通过文件头 magic bytes 识别引擎（xlsx→openpyxl，xls→xlrd），并在后续读取（表头/明细）保持同一 engine。
  - 如何避免/加固：
    - 不通过“改后缀名”转换格式；必须使用 Excel/WPS 的“另存为”。

  ## 11) 2026-03-06：各模块上传/选Sheet/读取逻辑不一致、且大表容易误以为卡死 → 全模块统一多选Sheet + 分阶段懒解析 + 读取进度

  - 模块/入口：主对账/回冲差异/快递费分析/汇总账单等所有 Excel 上传入口
  - 现象：
    - 不同模块的 Sheet 选择方式不一致（有的只能单选），用户学习成本高。
    - 大工作簿读取阶段无反馈，容易被误以为页面卡死。
  - 原因：
    - 历史上不同模块各自实现上传与读取，未复用同一套“多选Sheet+懒解析+进度提示”的策略。
  - 解决办法：
    - 统一上传解析口径：同一工作簿支持多选 Sheet。
    - 统一性能策略：Sheet名 → 表头（nrows=0）→ 明细按需读取（必要时 usecols）。
    - 统一用户反馈：在点击执行/生成时显示“正在读取 文件::Sheet（i/N）”与进度条。
  - 如何避免/加固：
    - 后续新增模块一律复用统一的 Excel 导入能力，不再单独写一套上传/读取逻辑。

  ## 12) 2026-03-07：拖拽字段映射弹窗报 ValueError（multi_containers 入参格式不符）

  - 模块/入口：主对账字段映射（`app.py` → `draggable_mapping_ui()` → `streamlit_sortables.sort_items()`）
  - 现象：
    - 打开“拖拽映射（模态对话框）”后页面报错：
      - `ValueError: items must be list[dict[str, Any]] if multi_containers is True.`
  - 原因：
    - `streamlit-sortables` 在 `multi_containers=True` 时，要求 `items` 必须为 `[{header: str, items: list}, ...]`。
    - 在某些 rerun / session_state 状态下，传参可能退化为 dict 或其它形态，触发组件的严格校验。
  - 解决办法：
    - 在 [app.py](../app.py) 增加入参清洗函数 `_sanitize_sortables_multi_containers()`：
      - 无论上游传入何种形态，进入组件前都转换为合法的 list-of-dict。
    - 同时关闭字段项的 hover 宽度/特效（禁用 transition/transform，透明边框 + border-box）以避免悬浮闪烁。
  - 如何避免/加固：
    - 对所有 `sort_items(multi_containers=True)` 调用点统一做入参规范化。
    - UI 样式避免“悬浮改变尺寸”的效果（会造成闪烁、影响拖拽手感）。
